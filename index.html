Motivation

  • Many decentralized web initiatives
  • Give control/sovereignty over data (Verstraete, 2022)
    • Where they store data
    • Who has access to it

Is it enough?

  • Divergence of interest or representation (Centralization of news even with social media, addictive design, sur-representation of certain languages) (Sevignani, 2013)
  • Utilization of users as unpaid sources of revenues even when data is anonymized (Mechant, 2021) (Sevignani, 2013)
  • Contribution to the centralization of wealth (Curran, 2016) (Sevignani, 2013) (Curran, 2016)

A more user-controlled way to power applications?

  • Make multiple engines provided by users execute discovery queries in web social network environments
  • A structural approach
  • The more users the lower the cost
  • More control over the design choices

Link Traversal Query Processing (LTQP)

  • Recursively looking up URIs to explore surrounding information, starting from a set of URIs called seed URIs (Hartig, 2016)
  • Predefined lookup policies with no input from the client (Hartig, 2016) (Hartig, 2012)
  • Difficulties; open-endedness of the web and query planning (Hartig, 2014)

Problem of completeness of LTQP in structured decentralized linked data environments

  • With more engines, more exploration of the domain
    • Search of non-overlapping section
    • Sharing of results
  • Structural biases of results
    • Sensibility of initial condition
    • Popularity biases

Problem of slow execution times of LTQP in structured decentralized linked data environments

  • We propose to use caching
  • Multiple similar, complementary, or identical queries
  • In LTQP caching: improves query completeness, can increase query execution time (Hartig, 2011)
    • Has not been evaluated in structured environments
  • The literature does not consider LTQP particularities: long execution time, multiple sources, difficulties in attaining completeness

Research questions

  1. Can we achieve better query result completeness and lower global query execution time?
  2. Does the CPU usage and the number of HTTP requests for each engine diminish?
  3. How can we prevent subdomain overlapping?
  4. How can we reduce query execution time using P2P caching?

Hypotheses

  1. Increasing the number of engines diminishes query execution time and increase the number of data sources explored.
  2. It is possible to create minimally overlapping subdomains without increasing query execution time.
  3. It is possible to create a P2P cache that decreases query execution time.

Collaborative SPARQL Querying

Using multiple agents to facilitate querying by diminishing the computation load of the execution or the discovery of data sources (Grall, 2020).

  • Rotating data sources in a P2P network; Snob (Grall, 2018)
    • Partial view of the network; random and profile-based
    • Results shared after the rotation
    • Possible to enhance completeness without visiting each data source

Collaborative SPARQL Querying

  • Division of the search space in communities: ColChain (Aebeloe, 2021)
    • Creation of deliberate communities
    • Querying own data source than the ones from known communities.
    • Collaboration in the division of the search space based on socially close information
  • Leveraging the social links between data sources (Shen, 2014) (Jin, 2006)

P2P caching in the context of the web

  • Mapping URLs in a distributed hash table (DHT): Squirrel (Iyer, 2002 )
  • Flower-CDN adds a mechanism to consider the locality of the node (El Dick, 2009)
  • Using an unstructured network with partial views; random and profiles based (behavioral locality): Behave (Frey, 2014)
  • CyCLaDEs adapt the concept of Behave to SPARQL queries of RDF documents (Folz, 2016)

Search domain division

  1. Divide seed URLs between query engines
  2. Adapt reachability criteria for each engine to the domain and query
  3. Use a global link queue and solution map

Divide seed URLs between query engines

Divide seed URLs between query engines

Advantage
  • Minimal communication between the engines
Disadvantages
  • Does not consider the overlaps
  • Potential loss of accuracy if the solutions are divided between multiple engines

Adapt reachability criteria for each engine to the domain and query

Adapt reachability criteria for each engine to the domain and query

Advantage
  • Minimal communication between the engines
Disadvantages
  • The criteria have to be changed depending on the type of query
  • Might not be suited to every query and dataset; data source not reachable, query not dividable by criteria, ...

Use a global link queue and solution map

Use a global link queue and solution map

Advantage
  • No special consideration or manipulation to get all the results
Disadvantage
  • Necessitate more communication between the engines

Metrics evaluated

  • Result accuracy
  • Query execution time
  • The ratio of the execution time and the communication time between the engines
  • Ability to access documents with a low number of edges
  • Overlapping of the search space
  • Query result arrival times

collaborative caching

  • Caching of URLs contributing to the query
  • intermediary join results
  • Three interpretations
    • Directory of (partial) results
    • Checkpoint for longer execution
    • Map of data sources to explore

Unstructured network where peers are clustered based on their behavior

Unstructured network where peers are clustered based on their behavior

Advantage
  • Faster lookup time
Disadvantage
  • Limited view of the network; depending on the organization of the network

Distributed Hash Table to find the cached element

Distributed Hash Table to find the cached element

Advantage
  • Ability to access the whole network
Disadvantages
  • Can be slow (log(n); n is the number of cached elements)
  • Not good for privacy, but there exist solutions (Nilizadeh, 2012) or it is possible to only share public data

Metrics evaluated

  • Cache access time
  • Cache miss and cache hit rates
  • Execution time reduction

Limitation and what else can be done

  • Mechanism for detecting similar queries and compatibility of the data publication with the collaborative mechanism; eg: Query containment (Afrati, 2019)
  • Fairness/"social contract" for sharing of computational power and cache
  • Measurement to know if it lowers the cost of social applications (it's hard to compare because it has plus values that are hard to quantify)

Conclusion

  • Building a Collaborative Link Traversal Query Processing in the Context of Structured Decentralized Environments
  • More complete queries by making engines collaborate
  • Faster queries by sharing cache information
  • Aim to make the creation of social applications more user-controlled
